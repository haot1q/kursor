Show and explain the current changes.

User-provided focus or scope hint (optional):
$ARGUMENTS

## Steps

1. If `$ARGUMENTS` looks like a commit / branch / range (e.g. `HEAD~3`, `main..feature/x`, `abc123`), use that as the diff target. Otherwise default to the working tree:
   - First check `git diff --cached --stat` (staged)
   - Then `git diff --stat` (unstaged)
   - If both are empty, fall back to `git log -1 -p` (last commit) and say so

2. Show a summary first:
   - Files changed, additions / deletions count
   - Group by area / package if the diff is large

3. Then explain what the changes do, **at the right level of abstraction**:
   - For a small diff: walk through it function by function
   - For a large diff: 1-paragraph overview + per-area bullets, only zooming into specific files if there's a non-obvious change
   - Distinguish behavior changes from refactors from cosmetic changes

4. Flag anything notable:
   - Public API surface changes
   - Test additions / removals
   - Config / schema changes
   - Possible regression risk (removed checks, changed defaults, broadened scope)

5. If asked, suggest a commit message (but do not commit unless the user explicitly says so — use `/commit` for that).

## Rules

- Do NOT modify files.
- Do NOT run any command other than read-only git commands.
- Keep the explanation proportional to the diff size — don't pad a one-line change with a five-paragraph essay.
