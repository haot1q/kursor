Generate a high-quality git commit for the current changes.

User-provided focus or hint (optional):
$ARGUMENTS

## Steps

1. Run `git status` to see staged vs unstaged changes.
2. Run `git diff --cached` if anything is staged, else `git diff` for unstaged.
3. Run `git log -n 5 --oneline` to learn this repo's commit message conventions (do they use Conventional Commits? imperative? subject + body? scopes? capitalization?).
4. **Match those conventions** in the message you write — don't impose your own style.
5. Decide whether to stage everything (`git add -A`) or only a subset. Default to staging only what's clearly related to the user's described change; if you can't tell, ask via the `question` tool.
6. Write the commit message:
   - Subject line: short, action-oriented (≤ 72 chars). Match the repo's tense / capitalization.
   - Body (only if non-trivial): brief WHY, not WHAT. The diff already shows WHAT.
   - No marketing language ("massive improvement", "amazing fix"). Be matter-of-fact.
   - Reference issues / PRs only if the user mentioned them.
7. Run `git commit -m "..."` (use a heredoc if the message has multiple lines):

   ```bash
   git commit -m "$(cat <<'EOF'
   subject

   body line 1
   body line 2
   EOF
   )"
   ```

8. Run `git status` afterwards to confirm the commit landed.
9. Report the new commit hash and message subject.

## Rules

- Do NOT push unless the user explicitly asks.
- Do NOT amend an existing commit unless the user explicitly asks.
- Do NOT commit files that look like secrets (`.env`, credentials, API keys). Warn the user if they're staged.
- Do NOT include AI-attribution lines like "Generated by …" unless the repo's existing commits use them.
- If pre-commit hooks fail, fix the issue and create a NEW commit. Never use `--no-verify` unless the user asks.
